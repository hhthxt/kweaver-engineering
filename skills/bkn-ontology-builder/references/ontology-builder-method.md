# BKN 本体建模方案生成方法

## 1. 输入成熟度判断

先判断是否具备生成本体建模方案的最小条件。至少需要以下任一类输入：

- PRD、BRD、业务流程说明或场景说明；
- 会议纪要、访谈转写或调研摘要；
- `BKN_Creator` 交接摘要；
- 系统材料、数据字典、样例数据，并且能看出业务目标和业务流程；
- 已有本体建模方案。

如果只有客户名称、行业方向、单句想法或技术目标，进入 `intake_or_readiness_check`，不要生成完整方案。

## 2. 场景优先抽取

不得从源表、字段、页面、报表、接口或 API 开始建模。先抽取：

- 业务目标；
- 用户角色；
- P0/P1 场景；
- 关键决策；
- 业务输入和业务输出；
- 当前流程和目标流程；
- 验收问题；
- 数据来源、系统边界和治理风险。

每个核心场景按以下顺序推导：

```text
业务目标
-> 被管理和判断的业务对象
-> 稳定业务关系
-> 事实 / 属性 / 指标
-> 可复用业务逻辑 / Skill 候选
-> 有副作用的业务行动
-> 运行状态和审计记录
-> 治理边界
-> 协同 / 工作空间 / 私有视角边界（仅在材料支持时）
-> 证据路径和验收映射
```

## 3. 对象建模纪律

核心对象是业务上被识别、管理、判断、追踪或协同的实体。不要因为某个词是名词就把它建成对象。

优先成为对象的内容：

- 有稳定身份；
- 有生命周期；
- 被业务角色持续管理或判断；
- 与多个场景或流程有关；
- 有状态、责任人、确认、调整、关闭、复盘或审计要求。

默认不成为对象的内容：

- 源表、报表行、查询结果行、页面组件；
- 临时计算结果；
- 角色名；
- 规则、口径、算法、提示词逻辑；
- 指标看板项；
- 只在一个动作里出现的中间变量。

如果结果类内容有确认、调整、分派、关闭、复盘等生命周期，可以作为对象候选，但必须标明确认状态和建模理由。

## 4. 关系建模纪律

关系必须服务业务路径，不是字段 join。

每条核心关系说明：

- 方向；
- 连接对象；
- 业务含义；
- 为什么需要；
- 服务哪个业务流程、状态推进、责任转移、影响路径或行动触发；
- 证据来源；
- 时间有效性、异常路径或适用范围；
- 确认状态。

关系方向容易误解时，在方案中显式说明，不让下游 Agent 猜测。

## 5. 指标、算子、解释逻辑、Skill 和 Action 边界

查询、计算、过滤、推理、解释等能力先判定为指标、算子、解释逻辑或治理策略。

只有当能力面向具体角色任务、具备输入输出契约，并需要被 Agent 编排调用时，才进入 Skill / Agent 应用契约。

只有具备业务副作用时才建模为 Action，例如：

- 改变对象状态；
- 写回系统；
- 创建审批、提醒、通知、工单或任务；
- 需要权限、审计、dry-run、回滚或人工确认。

查询类能力不得建模为 Action。

## 6. 稳定事实与运行状态分离

稳定事实不应被一次性事件污染。

推荐做法：

- 稳定对象保存长期身份、主数据和相对稳定事实；
- 事件影响、提醒、执行结果进入运行状态、事件记录、行动执行记录或审计对象；
- 方案说明事实变化、人工确认和 Action 执行结果分别如何影响对象状态；
- 对象生命周期、状态机和审计记录在业务运行层中说明。

## 7. 业务运行层写法

业务运行层是方案层表达，不等于 BKN 动力层。它回答“这些对象、关系、逻辑和行动如何进入真实业务运行”。

按材料适用性说明：

- 对象生命周期：创建、确认、调整、失效、关闭、复盘；
- 工作流入口：由谁在什么业务节点触发；
- 用户任务：谁查看、判断、确认、分派或执行；
- 状态变化：哪些来自事实变化、人工确认、Action 执行结果；
- 行动触发：哪些判断触发审批、写回、通知、提醒、工单或复核；
- 异常处理：数据不足、规则冲突、权限不足、执行失败；
- 反馈闭环：执行结果、业务反馈和复盘如何影响本体、规则或私有判断。

## 8. 业务证据链与可解释性

关键结论必须能解释依据。方案说明：

- 依赖哪些对象、关系、指标、算子或规则；
- 证据来自哪些来源材料、系统数据、人工确认或行动记录；
- 推理路径如何展示给业务用户；
- 数据不足、覆盖缺口或拒绝自动化时如何解释；
- 是否需要后续对接 TraceAI / Eval。

当前阶段只定义业务证据链、验收与留痕要求，不单独设计 Trace / Eval 模型。

## 9. 章节裁剪

不是所有项目都需要 Skill、Action、工作空间、协同机制、私有视角、TraceAI / Eval 对接或跨行业扩展。裁剪时要说明为什么不适用，避免漏建。

保留章节时，必须有材料依据或明确业务假设。没有材料支撑的内容应放入“建模候选”或“待业务确认”，不能写成已确认事实。

## 10. Verifier Rubric

`hard_stop`：

- 未列本轮输入来源；
- 未执行输入归档或登记，且未说明影响；
- 从源表、字段、页面、报表或接口开始建模，导致业务目标缺失；
- 把方案写成 `.bkn`、schema、平台操作说明或 BKN_Creator 强制门禁；
- 大量候选内容被写成已确认事实；
- 稳定事实与运行状态混在一起，影响核心业务判断；
- 查询类能力被建模为 Action，且会误导下游实现。

`major`：

- 核心对象缺少业务定义、设计理由、边界或建错风险；
- 关系缺少方向、业务含义、证据来源或使用场景；
- 指标、算子、Skill、Agent 和 Action 边界不清；
- 业务运行层缺少生命周期、状态推进、人工确认或反馈闭环；
- 关键判断无法追溯到来源材料、验收问题、对象、关系或规则；
- 协同、私有视角或跨场景复用被强行引入或遗漏。

`minor`：

- 表述不够业务化；
- 章节顺序或表格字段不完整但不影响判断；
- 个别待确认问题未归类；
- Final Gate 摘要不够清晰。

Verifier 输出必须独立成文，不直接改候选方案。若在项目目录落盘，使用 `assets/verifier-findings-template.md`，命名为 `<项目名>-本体建模方案 vX.Y-verifier-findings.md`。

## 11. 修订规则

Reviser 只根据 findings 定向修订：

1. 先处理 `hard_stop`；
2. 再处理 `major`；
3. 不删除已确认业务事实，除非确认状态错误；
4. 不把候选项改写成已确认内容；
5. 修订后重新执行 Verifier；
6. 最多自动修订 2 轮。

## 12. Final Gate 口径

`pass`：无 hard stop，major 已处理或能解释为不影响业务评审。

`warn`：无 hard stop，但仍有关键待确认；输出 Candidate 方案、待确认问题和下一步建议。

`fail`：hard stop 未清除；不声明方案可评审，输出未通过原因和补充材料清单。

Final Gate 输出必须独立成文。若在项目目录落盘，使用 `assets/final-gate-template.md`，命名为 `<项目名>-本体建模方案 vX.Y-final-gate.md`。

## 13. 标准 Harness 产物

正式 `scheme_mode` 或 `refine_mode` 写入项目目录时，保持以下文件边界：

| 环节 | 文件 | 是否必需 |
|---|---|---|
| Archive / Context | `inputs/round-XX/source-manifest.md` | 必需 |
| Generator | `<项目名>-本体建模方案 vX.Y-candidate.md` | 必需 |
| Verifier | `<项目名>-本体建模方案 vX.Y-verifier-findings.md` | 必需 |
| Reviser | `<项目名>-本体建模方案 vX.Y-revised-candidate.md` | 仅 findings 需要修订时 |
| Final Gate | `<项目名>-本体建模方案 vX.Y-final-gate.md` | 必需 |
| Final Output | `<项目名>-本体建模方案 vX.Y.md` 或保持 `candidate` 文件 | 必需 |

如果 Final Gate 为 `warn`，最终方案文件名应保留 `candidate` 或在文档状态中明确 `Candidate`，不得改写成 `Final`。

开发测试可额外输出 `*-harness-run-report.md`，用于记录一次样例运行覆盖了哪些环节和发现了哪些 Skill 资产问题。该文件不属于业务交付链路，不能替代 Verifier findings 或 Final Gate。
