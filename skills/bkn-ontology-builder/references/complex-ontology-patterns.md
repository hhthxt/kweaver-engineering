# 复杂本体场景抽象模式

本文件沉淀复杂业务场景中的通用建模模式和反模式，不包含客户专有 ID、平台实例、人员、敏感数据或不可公开业务细节。

## 1. 方案不是表结构映射

本体建模方案不能把本体做成数据库镜像。源表只是证据和数据来源，不是本体边界。

推荐模式：

- 按身份、生命周期和追踪要求分别建模稳定对象、事件 / 状态记录和审计记录；
- 用关系表达业务路径、状态推进、责任转移、影响路径或行动触发；
- 先识别指标、算子和解释逻辑，再在角色任务层形成 Skill / Agent 应用契约；
- 用 Action 表达有副作用的业务操作。

反模式：

- 一张源表对应一个对象；
- 一个外键或 SQL join 对应一条业务关系；
- 所有业务逻辑都塞进 Agent prompt；
- 查询、检索或解释类能力被建模为 Action。

## 2. 稳定事实与运行状态分离

稳定事实不应被单次事件污染。

示例原则：

- 长期供应能力、组织关系、产品结构、客户关系等是稳定事实；
- 停产、断供、涨价、舆情、异常订单、审批失败等事件造成的影响进入运行状态或审计对象；
- 事件结果应通过事件日志、提醒、状态记录或流程记录追溯。

## 3. Skill 与 Action 分工

查询、计算、过滤、推理、解释等能力先判定为指标、算子、解释逻辑或治理策略。

只有当这些能力面向具体角色任务、具备输入输出契约并需要被 Agent 编排调用时，才形成 Skill / Agent 应用契约。

只有具备业务副作用时才建模为 Action：

- 改变对象状态；
- 写回系统；
- 创建审批、提醒、通知或任务；
- 需要权限、审计、dry-run、回滚或人工确认。

## 4. 关系治理

关系必须服务业务路径，不能只是字段连接。

关系说明要覆盖：

- 方向；
- 业务含义；
- 为什么需要；
- 证据来源；
- 服务哪个业务流程、状态推进、责任转移、影响路径或行动触发；
- 时间有效性、异常路径或适用范围；
- 快捷语义关系是否需要底层证据对象支撑。

当关系方向容易误解时，应在方案中显式说明，并把规则沉淀到 Skill 或治理说明中。

## 5. 共享 Base 与私有判断隔离

当材料涉及多团队协作、共享底座或私有视角时：

- 共享事实进入 Base 本体；
- 分析师假设、团队规则、私有标签、关注列表和主观判断进入私有工作空间配置或标注层；
- 私有判断不得污染共享事实；
- 方案说明私有判断的生命周期、可见性、冲突处理，以及何时经确认晋升为共享 Base 事实或规则。

如果材料不涉及协作或私有视角，本节可省略，但 Verifier 应判断为什么不适用。

## 6. 设计理由是必填项

每组重要对象和关系后都需要设计理由。不要只输出裸对象清单。

设计理由建议回答：

- 该对象或关系服务哪个业务目标；
- 为什么不能只作为属性、指标、结果或临时状态；
- 如果不建模会影响哪个判断、协同或行动闭环；
- 建错会带来什么风险；
- 该设计是否已确认，还是建模候选。
